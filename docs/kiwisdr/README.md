# KiwiSDR 共享电台源：探测结论与字段解释（阶段 0）

相关素材：
- KiwiSDR 2 产品图：见项目内 [assets/kiwisdr2.jpg](file:///Users/mac/Documents/trae_projects/EchoFist/assets/kiwisdr2.jpg)

## 1. 设备角色（对 EchoFist 的意义）

KiwiSDR（含 KiwiSDR 2）更像“网络版短波接收机服务器”：把本地天线收到的 10 kHz–30 MHz 信号，通过网页/网络接口开放给远端用户使用，并支持多用户并发连接。

对 EchoFist 来说，KiwiSDR 是“共享电台源”。我们的目标不是追求生产级 SLA，而是用**礼貌、温和、可中止**的方式建立“可达性/可用性/稳定性”的长期记忆，服务于 CW 监听、盲解与自动化 QSO。

## 2. 探测接口与现实约束（业余/消费级特性）

共享电台生态多为业余爱好者部署，设备与网络条件差异很大：

- “取不到就是取不到”是常态：DNS 抖动、家庭宽带 NAT、动态域名、低功耗设备负载、站点限流等都会导致探测超时或返回不完整信息。
- 因此探测口径应采用“最佳努力”：能取到就取到，取不到不强求，把结果作为质量信号写入记忆即可。

## 3. `/VER` 返回字段解释（4 个字段）

我们在扫描中观察到 `/VER` 常见返回形如：

```json
{"maj":1,"min":841,"ts":4611686107326705383,"sp":0}
```

字段含义归纳如下（以“可验证行为”为优先）：

### 3.1 `maj`

- 主版本号（major）。
- 用途：粗粒度兼容性判断（大版本不同可能行为不同）。

### 3.2 `min`

- 次版本/构建号（minor/build）。
- 用途：与 `/status` 的 `sw_version=KiwiSDR_v1.xxx` 通常可对应（例如 `maj=1, min=841` 对应 `v1.841`）。

### 3.3 `ts`

- 服务器侧的“单调递增时间基准/计数器”（不是 Unix epoch）。
- 可验证行为：对同一台服务器连续请求 `/VER`，`ts` 会持续增加。
- 用途建议：
  - 作为“服务器时间基准在跑/是否重启跳变”的弱信号；
  - 不建议直接转成人类时间显示或做跨设备绝对对齐。

### 3.4 `sp`

- 当前仅能做“基于观测的推断”：其值通常为 0/1，表现为某类能力/配置档位标志。
- 观测关联：`sp` 与 `/status` 中的 `mode`（如 `rx8_wf2` vs `rx4_wf4`）和 `users_max`（容量上限）可能存在相关性，但缺少上游协议/源码证据，暂不作硬性结论。
- 用途建议：先作为“软特征”入库做统计，不作为硬淘汰条件；容量/拥塞优先看 `/status` 的 `users/users_max`。

## 4. 关键工程结论：`/VER` 可能是 JSON，但 Content-Type 不是 JSON

在实际服务器上我们观测到：

- `GET /VER` 返回 `status=200`，`content-type=text/plain`，但 body 仍是 JSON 文本（例如 `{"maj":1,"min":841,"ts":...,"sp":0}`）。
- 这会导致部分 HTTP 客户端（例如 aiohttp 的 `resp.json()` 默认策略）因为 mimetype 不匹配而拒绝解析，抛出 `ContentTypeError`。

因此对于 EchoFist 的探测逻辑：

- 应把 `/VER` 解析视为“最佳努力”：
  - 先以 `status==200` 作为 HTTP 可用的信号；
  - 再尝试以 `text + json.loads()` 的方式解析字段（不要强依赖 `application/json`）；
  - 解析失败不应否定 HTTP 可达性，只影响“字段可用度”。

## 5. 阶段 0 的“记忆基线”定义（当前可落地）

阶段 0 的目标是“可观测性闭环”，重点不在“更聪明”，而在“可审计、可回溯、可解释”：

- `scan_runs`：每次扫描形成批次记录（参数、来源、起止、摘要）。
- `scan_history`：每个源记录探测明细（run_id 关联、阶段耗时、error_kind、http_ok 等）。
- `sources`：聚合状态用于排序与淘汰（评分、成功/失败、连败、最近延迟、最近错误）。

这套基线为后续阶段 1 的“礼貌调度与退避”提供必要的数据基础。

## 6. 模块职责与数据模型（从代码内 docstring 整理）

为避免“文档与代码混在一起”，以下内容以文档为准，代码仅保留最小模块说明。

### 6.1 模块职责（对外消费能力）

- 按策略挑选候选源（过滤黑名单/失效名单、控制复检间隔）
- 批量并发探测并写回注册表（形成跨运行/跨日记忆）
- 提供维护能力（淘汰、过期清理、名单管理、单源洞察）

### 6.2 SQLite 表与语义

- `sources`：源主表（启用状态、累计成功/失败、最近延迟、综合评分等）
- `scan_runs`：探测批次（参数、来源、起止时间、运行摘要，用于审计与排障）
- `scan_history`：探测明细（run_id 关联 scan_runs、阶段耗时、error_kind、http_ok 等）
- `monitor_events`：运行期事件（reconnect/switch），用于对“稳定性”做惩罚
- `blocked_sources`：失效名单/黑名单（可选过期时间、原因、命中次数）

### 6.3 名单语义

- `blacklist`：明确拒绝（默认不过期）；写入时会强制 `sources.enabled=0`
- `invalid`：暂时隔离（支持 TTL）；常由 prune 自动写入，过期后允许重新探测

### 6.4 淘汰与清理机制（概念层）

- 基于连续失败、最小样本数、最小成功率触发 `enabled=0`（策略由配置项控制）
- prune 触发的禁用会同步写入 `invalid` 名单（TTL），用于“冷却隔离”
- 长期不活跃且已禁用的源支持清理（避免注册表无界增长）

## 7. 阶段性推进计划（从代码内 docstring 整理）

### 7.1 阶段 0：可观测性闭环（已启动）

- 每次探测建立 `scan_runs` 批次记录，并让 `scan_history` 关联 `run_id`
- 统一错误分类（`error_kind`）与基础耗时字段（`tcp_ms/http_ms/total_ms`）
- 观测：单次运行耗时分布、ok/fail 比例、错误类型分布、尾延迟

### 7.2 阶段 1：礼貌调度与退避

- 在 `sources` 落库 `cooldown_until_ts/next_probe_ts/backoff_level`
- 引入“单源退避 + 全局日预算 + 抖动”的慢扫策略，避免形成高频特征
- 观测：单位时间探测次数、重复探测率、冷却命中率、失败后重试间隔是否收敛

### 7.3 阶段 2：质量维度扩展

- 补充拥塞/满员信号（`users/max_users`）与运行期音频健康信号
- 将总分拆分为可解释子分（reachability/stability/congestion/audio）
- 观测：monitor 成功率、切台次数、音频中断率与评分一致性

### 7.4 阶段 3：历史留存与聚合

- `scan_history` 做“短期明细 + 长期按日聚合”，用聚合表支撑趋势与淘汰决策
- 观测：库体积增长曲线、查询耗时、淘汰误伤率

## 8. 安全默认值建议（礼貌优先）

- 并发建议 3-5（硬上限 10），limit 建议 ≤ 200；`verify_http` 可按场景关闭以降低额外请求
- 对公共资源始终优先“慢、可中止、可解释”，把预算与退避固化为长期机制

## 9. 阶段 0 全量扫描实测（记忆基线样例）

以下为一次“从干净沙盒开始”的全量基线扫描摘要（用于验证阶段 0 是否能跑通、以及 `/VER` 的可用性口径）：

- registry：`/tmp/kiwi_sources_test.sqlite3`
- 候选源：795（来自 `/tmp/kiwi_servers.txt`）
- 探测参数：`concurrency=3`，`timeout=1.2s`，`verify_http=True`，`post_delay_ms=200`

结果摘要：

- TCP 可达：`ok=483`，`fail=312`
- 平均 TCP 延迟（ok 样本）：`avg_latency_ms=534.4`
- `/VER` 统计：`http_200=249`，`json_ok=249`，`non_json_or_fail=0`

`/VER` 样本（均为 `status=200` 且 `content-type=text/plain`，body 为 JSON 文本）：

- `{"maj":1,"min":841,"ts":4611686129743754789,"sp":0}`
- `{"maj":1,"min":841,"ts":4611686107284995463,"sp":0}`
- `{"maj":1,"min":840,"ts":4611686786251944360,"sp":0}`

对工程口径的影响：

- `/VER` 在实测中表现为“body 是 JSON，但 mimetype 不是 application/json”，因此解析应采用“最佳努力”，不要把 mimetype 不匹配视为失败。
- `min` 可用于粗粒度区分软件版本（例如 840/841），但不应作为淘汰硬条件。
