# EchoFist 项目规则与约束

## 1. 项目概述

EchoFist 是一个面向业余无线电（HAM）爱好者的跨平台 AI 辅助等幅电报（CW）通讯软件。项目遵循"极客精神，文本为王"的理念，专注于功能与性能。

## 2. 开发环境与工具

### 2.1 Python 版本
- **最低版本**: Python 3.10
- **推荐版本**: Python 3.12
- **虚拟环境**: 必须使用虚拟环境（推荐使用 `venv`）

### 2.2 包管理
- **包管理器**: 使用 `pip` 进行依赖管理
- **依赖文件**: 
  - `requirements.txt` - 核心运行时依赖
  - `requirements-dev.txt` - 开发依赖
  - `pyproject.toml` - 项目配置和元数据

### 2.3 开发工具
- **代码格式化**: Black (行宽 88 字符)
- **代码检查**: Ruff (集成 Flake8、isort 等)
- **类型检查**: mypy (严格模式)
- **测试框架**: pytest + pytest-asyncio
- **代码提交**: pre-commit hooks

## 3. 代码规范

### 3.1 Python 编码规范
- **PEP 8**: 严格遵守 PEP 8 规范
- **格式化**: 使用 Black 风格，行宽 88 字符
- **类型注解**: 所有函数必须包含完整的 Python Type Hints
- **静态检查**: 基于 `basedpyright` 的最高等级类型检查（all）
- **命名规范**:
  - 变量名、函数名: `snake_case`
  - 类名: `PascalCase`
  - 常量: `UPPER_SNAKE_CASE`
  - 私有成员: `_private_member`
  - 特殊方法: `__special_method__`

### 3.2 语言使用规范
- **对话与文档**: 使用中文
- **代码注释**: 使用中文
- **代码命名**: 使用英文
- **文档输出**: 使用中文，符合 Markdown 语法

### 3.3 代码质量要求
- **无冗余代码**: 使用 Flake8 确保代码简洁
- **无潜在风险**: 使用 BugBear 检查潜在问题
- **导入排序**: 使用 isort 自动排序导入
- **复杂度控制**: 函数复杂度不超过 15，类复杂度不超过 30

### 4.2 模块导入约束
- **绝对导入**: 优先使用绝对导入
- **相对导入**: 仅在包内部使用相对导入
- **导入顺序**:
  1. 标准库导入
  2. 第三方库导入
  3. 本地应用/库导入
- **避免循环导入**: 使用依赖注入或重构设计

## 5. 功能边界约束

### 5.1 核心功能边界
- **音频处理**: 专注于 CW 信号处理，不处理语音或数字模式
- **网络通信**: 支持 KiwiSDR 协议，不实现其他 SDR 协议
- **数据存储**: 使用 SQLite 作为默认数据库，支持 PostgreSQL 作为可选
- **用户界面**: 基于文本的 TUI，不提供图形界面

### 5.2 性能约束
- **实时性**: 音频处理延迟 < 100ms
- **内存使用**: 单次处理内存占用 < 256MB
- **CPU 使用**: 单核 CPU 占用率 < 80%
- **网络延迟**: KiwiSDR 连接延迟 < 500ms

### 5.3 兼容性约束
- **操作系统**: 支持 Linux, macOS, Windows
- **Python 版本**: 支持 Python 3.10+
- **硬件要求**: 支持普通声卡，不依赖特殊硬件
- **网络环境**: 支持普通家庭宽带网络

## 6. 测试与质量保证

### 6.1 测试覆盖率要求
- **核心功能**: 测试覆盖率 ≥ 90%
- **关键路径**: 测试覆盖率 100%
- **边缘情况**: 必须包含边界条件测试
- **性能测试**: 包含基准测试和压力测试

### 6.2 测试类型
- **单元测试**: 测试单个函数或类
- **集成测试**: 测试模块间交互
- **端到端测试**: 测试完整工作流程
- **异步测试**: 使用 pytest-asyncio 测试异步代码

### 6.3 测试运行
```bash
# 运行所有测试
pytest

# 运行测试并生成覆盖率报告
pytest --cov=echofist --cov-report=html

# 运行特定测试模块
pytest tests/test_audio_processor.py
```

## 7. 部署与发布

### 7.1 版本管理
- **语义化版本**: 遵循 Semantic Versioning 2.0.0
- **版本号格式**: `MAJOR.MINOR.PATCH`
- **发布周期**: 每月发布一个次要版本

### 7.2 打包要求
- **打包工具**: 使用 setuptools + wheel
- **平台支持**: 提供跨平台包
- **依赖管理**: 明确指定依赖版本范围

### 7.3 发布流程
1. **代码审查**: 所有更改必须经过代码审查
2. **测试通过**: 所有测试必须通过
3. **版本更新**: 更新版本号和变更日志
4. **打包发布**: 构建并发布包到 PyPI
5. **文档更新**: 更新用户文档和 API 文档

## 8. 安全与合规

### 8.1 安全要求
- **无硬编码密钥**: 禁止在代码中硬编码密钥
- **输入验证**: 所有用户输入必须验证
- **错误处理**: 错误信息不泄露敏感信息
- **依赖安全**: 定期更新依赖，修复安全漏洞

### 8.2 合规要求
- **业余无线电法规**: 遵守当地业余无线电法规
- **频率使用**: 仅在允许的频率范围内操作
- **呼号使用**: 必须使用合法呼号进行通联
- **数据隐私**: 保护用户通联记录的隐私

## 9. 文档要求

### 9.1 代码文档
- **函数文档**: 所有公共函数必须有文档字符串
- **类文档**: 所有类必须有文档字符串
- **模块文档**: 每个模块必须有模块级文档
- **类型文档**: 复杂类型必须有类型注解说明

### 9.2 用户文档
- **安装指南**: 详细的安装步骤
- **使用教程**: 从入门到进阶的教程
- **API 文档**: 完整的 API 参考
- **故障排除**: 常见问题解决方案

### 9.3 开发文档
- **架构设计**: 系统架构和设计决策
- **开发指南**: 开发环境设置和开发流程
- **贡献指南**: 贡献代码的流程和要求
- **发布流程**: 版本发布的具体步骤

## 10. 变更管理

### 10.1 变更流程
1. **创建 Issue**: 描述问题或功能需求
2. **分支开发**: 从主分支创建功能分支
3. **代码提交**: 提交代码并添加测试
4. **代码审查**: 至少一名审查者批准
5. **合并请求**: 合并到主分支
6. **版本发布**: 发布新版本

### 10.2 变更记录
- **变更日志**: 维护详细的变更日志
- **版本历史**: 记录每个版本的变更内容
- **向后兼容**: 尽可能保持向后兼容性
- **废弃警告**: 提前通知废弃的功能

## 11. 异常处理与日志

### 11.1 异常处理原则
- **明确异常**: 使用具体的异常类型
- **适当处理**: 在适当层级处理异常
- **资源清理**: 确保资源正确释放
- **错误恢复**: 提供错误恢复机制

### 11.2 日志规范
- **日志级别**: DEBUG, INFO, WARNING, ERROR, CRITICAL
- **日志格式**: 结构化日志，包含时间戳、级别、模块、消息
- **敏感信息**: 日志中不记录敏感信息
- **日志轮转**: 自动轮转日志文件，防止过大

## 12. 性能监控

### 12.1 监控指标
- **处理延迟**: 音频处理各阶段延迟
- **解码准确率**: CW 解码准确率统计
- **内存使用**: 运行时内存使用情况
- **CPU 使用**: 各模块 CPU 使用情况

### 12.2 性能优化
- **性能基准**: 建立性能基准测试
- **瓶颈分析**: 定期分析性能瓶颈
- **优化验证**: 优化后验证性能提升
- **回归测试**: 防止性能回归
